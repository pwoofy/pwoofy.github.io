<h1>AES ECB: Why do we even still have this?</h1>

Imagine this. You open a crypto challenge, check the provided script, and you see `AES.MODE_ECB`. The immediate instinct is to smile, crack your knuckles, and take your free points.

Electronic Codebook (ECB) is the simplest, most naive mode of AES encryption. It takes your plaintext, chops it into 16-byte blocks, and encrypts each block independently using the key. 

Because blocks don't interact with each other, **identical plaintext blocks always produce identical ciphertext blocks.** This is the fatal flaw that created the famous "ECB Penguin" image.


<h2>Type 1: The FREE!!! Version (Cut and Paste)</h2>

Challenge description:
> I made a super secure login token using AES! Normal users get normal tokens. But if you somehow log in as an admin, you get the flag. Mwehehehe.

<div class="sage">
  <script type="text/x-sage">
import os
import operator

KEY = os.urandom(16)

def pad(data, block_size):
    padding_len = block_size - (len(data) % block_size)
    return data + bytes([padding_len] * padding_len)

def unpad(data):
    padding_len = data[-1]
    return data[:-padding_len]

def ecb_encrypt(plaintext, key):
    padded = pad(plaintext, 16)
    return b"".join([bytes([operator.xor(b, k) for b, k in zip(padded[i:i+16], key)]) for i in range(0, len(padded), 16)])

def ecb_decrypt(ciphertext, key):
    padded_plaintext = b"".join([bytes([operator.xor(b, k) for b, k in zip(ciphertext[i:i+16], key)]) for i in range(0, len(ciphertext), 16)])
    return unpad(padded_plaintext)

def generate_token(username):
    data = f"user:{username}role:guest".encode()
    return ecb_encrypt(data, KEY)

token_bytes = generate_token("meow")
print(f"Token: {token_bytes.hex()}")

decrypted_data = ecb_decrypt(token_bytes, KEY)
print(f"Decrypted: {decrypted_data}")

<details>
  <summary><b>💡 Interactive: How do we actually cut and paste? (Click to reveal)</b></summary>

  Let's break down exactly how AES ECB chops up our string. We know the server builds the token like this: <br>
  <code>"user:" + username + "role:guest"</code><br><br>

  Our goal is to create a token that ends with <code>role:admin</code>. Since we don't have the key, we can't create a credential that has the role of admin, so we have to trick the server into giving the admin credential ourselves.

  <br>
  First, we need to know where the blocks start and end. "user:" is exactly 5 bytes long. If we send a username of 11 "A"s, the first block is perfectly full (16 bytes). <br><br>
  
  Let's look at what the server does if we just send <code>AAAAAAAAAAA</code>:<br>
  <b>Block 1:</b> <code>user:AAAAAAAAAAA</code> (16 bytes)<br>
  <b>Block 2:</b> <code>role:guest\x06\x06\x06\x06\x06\x06</code> (16 bytes, with PKCS#7 padding)<br><br>

  <i>(Note: AES requires data to be a multiple of 16. Since "role:guest" is 10 bytes, the server automatically adds six "06" bytes to fill the block. This is called PKCS#7 padding.)</i><br><br>

  We need a ciphertext block that decrypts exactly to <code>role:admin</code> plus the correct padding. "role:admin" is 10 bytes long, so it also needs six "06" bytes.<br><br>
  
  How do we get the server to encrypt this? We just put it in our username! We will send 11 "A"s (to fill Block 1), and then our payload (to perfectly fill Block 2).<br><br>

  Let's send this username: <code>AAAAAAAAAAArole:admin\x06\x06\x06\x06\x06\x06</code><br><br>

  Here is how the server chops up and encrypts our input:<br>
  <b>Block 1:</b> <code>user:AAAAAAAAAAA</code> (16 bytes)<br>
  <b>Block 2:</b> <code>role:admin\x06\x06\x06\x06\x06\x06</code> (16 bytes) <i><-- WE WANT THIS!</i><br>
  <b>Block 3:</b> <code>role:guest\x06\x06\x06\x06\x06\x06</code> (16 bytes padded)<br><br>

  When the server gives us the long hex string for that token, it is just Block 1 + Block 2 + Block 3 stitched together.<br><br>
  
  Because ECB doesn't link the blocks, we can just throw away Block 3! If we submit only the ciphertext for Block 1 + Block 2, the server will decrypt it and see:<br>
  <code>user:AAAAAAAAAAArole:admin</code><br><br>

  Yippeee!!
</details>

<h2>Type 2: The Free(n't) flag... (The Oracle)</h2>

Challenge description:
> You can't forge blocks if I don't give you the role! I appended the flag directly to your input and encrypted it. Good luck guessing it now!

<div class="sage">
  <script type="text/x-sage">
from Crypto.Cipher import AES
from Crypto.Util.Padding import pad
import os

KEY = os.urandom(16)
FLAG = b"crypto{0rac1e_m4g1c}"

def encrypt_oracle(user_input):
    # The server appends the secret flag to WHATEVER you send it
    plaintext = user_input + FLAG
    cipher = AES.new(KEY, AES.MODE_ECB)
    return cipher.encrypt(pad(plaintext, 16))

print("Oracle Output for 'test':")
print(encrypt_oracle(b"test").hex())
  </script>
</div>

<details>
  <summary><b>💡 Interactive: How do we do this? (Click to reveal)</b></summary>
  This is called a <b>Byte-at-a-Time Chosen Plaintext Attack</b>. By controlling the length of our input, we can force the server to spill the flag one letter at a time.
  <br><br>

  <b>The Block Shift</b><br>
  Let's assume the block size is 16 bytes. What happens if we send exactly 15 bytes of junk, like <code>"AAAAAAAAAAAAAAA"</code>?<br><br>
  The server appends the flag to our input. So the first 16-byte block the server encrypts looks like this:<br>
  <code>[ A A A A A A A A A A A A A A A F ]</code><br><br>

  
  
  That <b>F</b> is the <i>first letter of the secret flag!</i> It got pulled into our block because our 15 'A's left exactly 1 byte of free space. <br><br>

  <b>The Oracle Dictionary</b><br>
  The server gives us the ciphertext for that block. We save it. Now, how do we figure out what that last letter is?<br><br>
  We just ask the server to encrypt everything! We send:<br>
  <code>AAAAAAAAAAAAAAAa</code><br>
  <code>AAAAAAAAAAAAAAAb</code><br>
  <code>AAAAAAAAAAAAAAAc</code><br><br>
  
  We iterate through the entire alphabet. Because ECB always encrypts identical blocks to identical ciphertexts, the exact moment our guess matches the server's ciphertext, we know we guessed the right letter! <br><br>

  <b>Moving Forward</b><br>
  Once we know the first letter (let's say it's <code>c</code>), we shrink our padding to 14 bytes: <code>AAAAAAAAAAAAAA</code>. <br>
  Now the block pulls in the first <i>two</i> letters of the flag: <code>[ A A A A A A A A A A A A A A c r ]</code>.<br><br>
  We already know the <code>c</code>, so we just iterate the last byte again until we find the <code>r</code>. We repeat this until the entire flag is dumped!
</details>

from Crypto.Cipher import AES
from Crypto.Util.Padding import pad
import os
import string

# --- 1. THE CTF SERVER ---
KEY = os.urandom(16)
FLAG = b"crypto{3cb_b4by_st3p5}"

def server_encrypt(user_input):
    plaintext = user_input + FLAG
    cipher = AES.new(KEY, AES.MODE_ECB)
    return cipher.encrypt(pad(plaintext, 16))


# --- 2. THE ATTACKER SCRIPT ---
print("--- Starting ECB Oracle Attack ---")
known_flag = b""
block_size = 16

# We'll just extract the first 15 bytes for this example
for i in range(1, 16):
    # Create the padding (15 'A's, then 14, then 13...)
    padding = b"A" * (block_size - i)
    
    # Get the target ciphertext block from the server
    target_block = server_encrypt(padding)[:16]
    
    # Iterate through printable characters to guess the byte
    for char in range(32, 127):
        guess = padding + known_flag + bytes([char])
        guess_block = server_encrypt(guess)[:16]
        
        # If the ciphertexts match, we found the letter!
        if guess_block == target_block:
            known_flag += bytes([char])
            print(f"[+] Byte found! Flag so far: {known_flag.decode()}")
            break

print(f"\n[!] Final Extracted Flag: {known_flag.decode()}")

<h1>Final Thoughts</h1>

If you can control the data going into an ECB cipher, you own the cipher. Whether it's cutting and pasting valid blocks to escalate privileges, or shifting a secret string byte-by-byte into a block you control, ECB's lack of diffusion is its ultimate downfall.
