<h1>To cbrt or not to cbrt?</h1>

Imagine this. You open a crypto challenge, you check the provided script, and you see `e = 3`. The immediate instinct is to just take the cube root of the ciphertext and call it a day. 

Sometimes the CTF author is feeling generous. Sometimes, they are actively setting a trap. Let's look at the difference.

<h2>Type 1: The FREE!!! Version</h2>

Challenge description:
> I encrypted my suuuuper duper secret flag. Good luck factoring my massive 1024-bit modulus! Mwehehehe.

<div class="sage">
  <script type="text/x-sage">

p = random_prime(2^512)
q = random_prime(2^512)
N = p * q
e = 3
flag = b"crypto{ooh}"
m = int.from_bytes(flag, 'big')
C = power_mod(m, e, N)
print(f"N = {N}")
print(f"C = {C}")
  </script>
</div>

<details>
  <summary><b>💡 Interactive: How do we solve this? (Click to reveal)</b></summary>
  Let's look at the textbook RSA encryption formula:
$$C \equiv m^e \pmod N$$

If the message $m$ is relatively short, and $e$ is only $3$, $m^3$ has a chance to be *smaller* than the massive modulus $N$. <br><br>

But what does this mean? Let's take an example.<br><br>

Let's say that you have a message. For simplicity's sake, we assume our message is a singular number. Let's say the number is 40. And if we take our p and q as a really large prime. For simplicity's sake, let's assume that $p*q$ is 1 million (1,000,000). What does this do?

Well, going back to the following formula:

$$C \equiv m^e \pmod N$$

Our ciphertext would be $40^3 \pmod {1000000}$, or $64000 \pmod {1000000}$. What does this mean?
  <br>
  Since the modulo didn't wrap around, the ciphertext $C$ is literally just the raw $m^3$. <br><br>
  You don't need to factor the modulus or find the private key. You literally just take the standard integer cube root of $C$ to recover the flag. Very easy.
</details>

<br>
<h3>Try it live!</h3>

<h2>Type 2: The Free(n't) flag...</h2>

Challenge description:
> I figured out the cube root trick, so I padded the message with some extra text at the front. Good luck now!
<br>
> Prefix: "i_pad_this_part_lol: "

<div class="sage">
  <script type="text/x-sage">
p = random_prime(2^512)
q = random_prime(2^512)
N = p * q
e = 3
prefix = b"i_pad_this_part_lol: "
flag = b"crypto{winner_but_i_pad_the_message}"
m = int.from_bytes(prefix + flag, 'big')
C = power_mod(m, e, N)
print(f"N = {N}")
print(f"C = {C}")
  </script>
</div>

<details>
  <summary><b>💡 Interactive: How do we do this? (Click to reveal)</b></summary>
  Let's go back to our formula:
$$C \equiv m^e \pmod N$$

By adding a known prefix, the author has made our total message $m$ much larger. To solve this, we define a polynomial where our unknown flag is $x$:
$$f(x) = (\text{prefix} \cdot 2^k + x)^3 - C \equiv 0 \pmod N$$

<b>The Modulo</b><br>
Let's shrink everything down to see the problem. Imagine our modulus is 323. Our known prefix is 10, and our secret PIN $x$ is 2. The total message is 12.<br><br>
$12^3 = 1728$. <br>
1728 modulo 323 is 113. So our intercepted ciphertext is $C = 113$.<br><br>

Our polynomial looks like this:
$$f(x) = (10 + x)^3 - 113 \equiv 0 \pmod{323}$$

If we expand that using basic algebra, we get:
$$f(x) = x^3 + 30x^2 + 300x + 887 \equiv 0 \pmod{323}$$

We can simplify 887 by taking modulo 323, leaving us with our final trapped polynomial:
$$f(x) = x^3 + 30x^2 + 300x + 241 \equiv 0 \pmod{323}$$

Why can't we just solve this like a high school algebra problem to find $x$? Because if you plug $x = 2$ into that equation using normal math, you get <b>969</b>. <br><br>
In Modulo 323, 969 is equal to 0 (because 969 divided by 323 is exactly 3). But a standard algebra solver doesn't know about the modulo! It just sees that the answer is 969, so it assumes $x = 2$ is the wrong answer. Without factoring $N$, we are stuck behind a mathematical brick wall.<br><br>

<b>The Loophole (Howgrave-Graham)</b><br>
Enter Don Coppersmith and a mathematical cheat code known as <i>Howgrave-Graham's Theorem</i>. This theorem states that if we have a polynomial equal to $0 \pmod N$, AND the overall "size" of the polynomial's coefficients is strictly less than $N$, then the modulo completely vanishes. The equation becomes $f(x) = 0$ over standard integers, which is trivial to solve!<br><br>

<b>Building the LLL Matrix manually</b><br>
To trigger this loophole, we need to build a <b>lattice</b>. You can think of a lattice as an infinite, multi-dimensional grid of dots. Every point on this grid is created by mixing and adding together a set of base vectors (which will be the rows of our matrix). <br><br>



Here is the genius part: we want LLL to find a new polynomial that evaluates to $0 \pmod N$. So, we seed our lattice matrix with a toolkit of basic polynomials that we <i>already know</i> equal $0 \pmod N$. 

Think about it:
<ul>
  <li>$N$ is a multiple of $N$, so mathematically $N \equiv 0 \pmod N$.</li>
  <li>$N \cdot x$ is just $N$ times something, so it also $\equiv 0 \pmod N$.</li>
  <li>$N \cdot x^2$ is also $\equiv 0 \pmod N$.</li>
  <li>And our intercepted polynomial $f(x)$ is exactly what we are trying to solve, so $f(x) \equiv 0 \pmod N$.</li>
</ul>

Because every row in our matrix equals 0 mod $N$, <i>every single point in the entire infinite lattice</i> represents a combination of these polynomials that also equals 0 mod $N$. We are just asking LLL to search this grid and find the point closest to the origin (the shortest vector). That shortest vector will be our new polynomial with tiny coefficients!<br><br>

<b>The Gamble ($X$)</b><br>
There is one problem: LLL is just a geometry algorithm. It doesn't know that $x$ is a variable that grows exponentially. To force LLL to respect that higher powers of $x$ are mathematically "heavier", we apply a gamble bound $X$ to the columns. $X$ is the absolute maximum size we think our secret $x$ could be. Let's gamble that $X = 5$.<br><br>

We build our 4x4 matrix row by row. Each column represents a power of $x$ ($x^0, x^1, x^2, x^3$), scaled by our $X$ gamble:
<ul>
  <li>Row 1 (The $N$ polynomial): 323, 0, 0, 0</li>
  <li>Row 2 (The $Nx$ polynomial): 0, $323 \cdot 5$, 0, 0 $\rightarrow$ <b>0, 1615, 0, 0</b></li>
  <li>Row 3 (The $Nx^2$ polynomial): 0, 0, $323 \cdot 5^2$, 0 $\rightarrow$ <b>0, 0, 8075, 0</b></li>
  <li>Row 4 (Our $f(x)$ polynomial): 241, $300 \cdot 5$, $30 \cdot 5^2$, $1 \cdot 5^3$ $\rightarrow$ <b>241, 1500, 750, 125</b></li>
</ul>

This is exactly what SageMath is doing in the background! But there is a catch. If you run LLL on this exact matrix with $X = 5$, <b>it fails</b>. The numbers in the matrix (the "volume") are too bloated, so LLL can't find a vector short enough to trigger the Howgrave-Graham loophole.<br><br>



If we reduce our gamble to $X = 3$, the volume shrinks just enough. We can actually run LLL on this raw matrix manually using this Sage script:



If we reduce our gamble to $X = 3$, the volume shrinks just enough. We can actually run LLL on this raw matrix manually using this Sage script:

<div class="sage">
  <script type="text/x-sage">
N = 323
X = 3

M = Matrix(ZZ, [
    [N,   0,       0,         0],
    [0,   N*X,     0,         0],
    [0,   0,       N*(X^2),   0],
    [241, 300*X,   30*(X^2),  1*(X^3)]
])

reduced_M = M.LLL()
sv = reduced_M[0]

coeffs = [
    sv[0] / (X^0),
    sv[1] / (X^1),
    sv[2] / (X^2),
    sv[3] / (X^3)
]

P.<x> = PolynomialRing(ZZ)
np = sum(coeffs[i] * (x**i) for i in range(4))
roots = np.roots()

print(f"PIN Code:: {roots[0][0]}")
  </script>
</div>

The modulo $N$ drops away, Howgrave-Graham's loophole opens, and LLL hands us the secret. When you use `.small_roots()` on a real CTF challenge, it is just automating this exact matrix construction (and expanding it to be much larger to handle bigger $X$ bounds).
</details>

<h3>Try it live!</h3>
The following block of code runs both the server generation and the LLL attacker script. (Solve script is written in SageMath)

<div class="sage">
  <script type="text/x-sage">
p = random_prime(2^512)
q = random_prime(2^512)
N = p * q
e = 3

prefix = b"i_pad_this_part_lol: "
flag = b"crypto{winner_but_i_pad_the_message}"

m_int = int.from_bytes(prefix + flag, byteorder='big')
C = power_mod(m_int, e, N)

print(f"N = {N}")
print(f"C = {C}")
print(f"Known Prefix: {prefix}")

k_bits = 288  
X_gamble = 2^k_bits

prefix_int = int.from_bytes(prefix, byteorder='big')
base_message = prefix_int * (2^k_bits)

P.<x> = PolynomialRing(Zmod(N))
f = (base_message + x)^e - C
f = f.monic()

print(f"Building lattice matrix and running LLL...")
roots = f.small_roots(X=X_gamble, epsilon=0.03)

if roots:
    secret_x = int(roots[0])
    recovered_bytes = secret_x.to_bytes((secret_x.bit_length() + 7) // 8, byteorder='big')
    print(f"\nflag: {recovered_bytes.decode()}")

  </script>
</div>

<h1>Final Thoughts</h1>

If $m^3 < N$, just cube root it and take your points. If $m^3 > N$ and you know part of the plaintext, you have to build the matrix and let LLL do the heavy lifting. Once you understand what the `small_roots` function is actually doing under the hood, these Crypto challenges become incredibly satisfying to solve, hehe :3c
